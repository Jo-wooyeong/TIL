## pull
```
특정 브랜치에서 작업내용을 가져올 때 사용
```

우선 작업 순서 리마인드
```
1. 내 브런치에서 작업 후 add, commit, push
2. 깃에서 머지
3. (상대방이 업데이트 한 브랜치) 협업 했던 작업을 통합하는 브랜치
```

### git pull 설정?
각 작업자가 같은 브랜치에 머지 후 내 브랜치에서 해당 내용을 pull받으려고 했는데 다음과 같은 에러가 나왔다.
![git-pull-error](/public/git-pull-error.png)

pull을 받을건데 rebase를 사용할 것인가 아닐것인가를 선택하라는 문구,

#### 여기서 `rebase`란?
merge는 갈라진 히스토리를 한번에 합치고 충돌을 한번에 해결합니다. 하지만 `rebase`는 내 커밋을 한장씩 옮겨 붙이는 과정이기 때문에 커밋 한만큼 충돌 해결해줘야합니다.<br />

장점은 히스토리가 한 줄이 된다. 이미지는 머지했을때 복잡해진 히스토리 선
![merge-history](/public/merge-history.png)

커밋량이 많으면 많을 수록 `rebase`할때 수동 처리해야하는 양이 많아지기 때문에 백/프론트 작업이라면 거의 겹치는 작업이 없으므로 `merge`하는게 정신건강에 좋다.<br />

그래도 같은 파일에서 충돌이 나면 한 번더 알려주니 안심!

### merge
`pull.rebase false` 설정을 완료하면 `pull` 받을 때 머지로 처리가 되는데, 머지 하고나면 `vi 편집기`가 나오는데 이때 `:q`하면 머지 완료

### push
머지를 받았다면 로컬 저장소에 업데이트가 된 것이기 때문에, 같은 이름의 원격브랜치에도 업데이트를 해줘야한다.
- `git add, commit, push` 실행
